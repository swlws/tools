# ai-commit · 语义化分组提交

把**暂存区**（`git add` 过的内容）按语义分组，每组生成一个规范 commit，经用户确认后逐组提交。

## 铁律

1. **只处理暂存区**：分析对象是 `git diff --cached`。工作区未暂存的改动一律不看、不 `git add`、不提交。
2. **不改动工作区**：绝不使用 `git reset --hard`、`git checkout --`、`git stash` 等可能触及工作区内容的命令。默认 `git reset`（`--mixed`）只退索引不动工作区，但本 skill 优先避免 reset（见下"部分暂存"处理）。
3. **先确认后提交**：展示分组方案，用户明确同意后才执行 `git commit`。

## 流程

### 1. 收集暂存内容
- `git diff --cached --stat` 看暂存了哪些文件
- `git diff --cached` 读具体改动做语义分析
- 若暂存区为空，告知用户"暂存区无内容"，结束

### 2. 检测"部分暂存"风险
同一文件可能既有已暂存、又有未暂存改动（部分 `git add`）。检测方式：比较 `git diff --cached --name-only` 与 `git diff --name-only` 的交集——交集中的文件即"部分暂存"。

- **若无部分暂存**：可安全按分组分别提交
- **若存在部分暂存**：分组提交时用 pathspec 只提交这些文件的暂存版本，避免把工作区未暂存部分带入。**永远不要** `git reset` 后 `git add <file>`（那会把工作区完整版本重新加入，违反铁律 2）

### 3. 语义分组
按「变更类型 + 涉及模块」把暂存文件分成若干逻辑组，每组对应一个提交：

- 类型：`feat`（新功能）/ `fix`（修复）/ `style`（样式/格式）/ `refactor`（重构）/ `docs`（文档）/ `chore`（杂项/配置）/ `test`
- 同类型且相关的文件归为一组；不相关的功能拆成不同组
- 每组拟定一条 commit message：中文、聚焦"为什么"

### 4. 展示并确认
分组后，在控制台按**固定格式**逐组输出方案。每组用分隔线卡片呈现，格式如下：

```
════════════════════
[组 {序号}/{总组数}] {type}
文件: {文件1}, {文件2}, ...
msg : {commit message}
════════════════════
```

要求：
- 分隔线用 20 个 `═`，每组上下各一条（相邻组共用中间那条即可）
- `[组 1/2]` 标明当前组序号与总组数
- `文件:` 列出该组全部文件，逗号分隔
- `msg :` 为拟定的完整 commit message（含 type 前缀）

输出全部分组卡片后，**用 `AskUserQuestion` 工具询问用户是否同意该分组提交方案**（而非仅等待自由文本回复）。用户在 Ask 中明确选择同意后才进入第 5 步；未同意不执行。

### 5. 逐组提交
确认后，对每组用 pathspec 精确提交，只作用于该组的暂存内容：

```bash
git commit -m "<message>" -- <该组文件1> <该组文件2>
```

用 `git commit -- <pathspec>` 直接提交指定文件的**已暂存**改动，天然只提交暂存版本、不影响工作区、也不影响其他组的暂存文件。全部提交后 `git status` 验证。

**提交完成后不执行 `git push`**（除非用户在本次对话中另外明确要求）。

## 边界

- 不 `push`（除非用户另外要求）
- 不修改、不新增暂存内容，只是把已暂存的分批提交
- 疑似含密钥的文件（.env、credentials 等）提交前警告用户
- 不使用 `--amend`、`--no-verify`（除非用户明确要求）
